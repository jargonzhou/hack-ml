# Information Retrieval

- 信息检索算法
```markdown
信息检索（Information Retrieval, IR）算法是现代搜索引擎、推荐系统以及大模型 RAG（检索增强生成）系统的底层基石。其核心任务是：从海量的非结构化文档库中，精准、高效地找出与用户输入查询（Query）最相关的核心信息。
信息检索算法的发展经历了三个核心世代，并在当下的 RAG 架构中走向了多路融合。
------------------------------
## 一、 经典代：基于字面匹配的稀疏检索（Sparse Retrieval）
这一代算法主要依赖词频统计和文本倒排索引（Inverted Index），核心是计算词语的字面重合度。

* TF-IDF（词频-逆文档频率）
* 原理：评估一个词对于一个文档集或一个语料库中的其中一份文档的重要程度。
   * 逻辑：字面词频（TF）越高越重要；但在所有文档中出现频率（DF）越高则越不重要（如“的”、“在”）。
* BM25（Best Matching 25）
* 地位：传统文本检索的行业标准，Elasticsearch 和 Lucene 的核心默认算法。
   * 升级点：在 TF-IDF 基础上引入了词频饱和度抑制（防止关键词堆砌作弊）和文档长度归一化（公平对待长短文本）。
* 特征总结：计算极快，对专有名词、产品型号、错误拼写极其精准；但无法理解同义词和上下文语境。
* 

------------------------------
## 二、 现代代：基于深度学习的稠密检索（Dense Retrieval）
伴随 BERT 等预训练大模型的发展，检索走入了“语义空间”。算法将文本转化为连续的高维稠密向量（Dense Embeddings）。
## 1. 向量化建模架构

* 双塔架构（Bi-Encoder）：
* 逻辑：文档端和查询端分别通过独立的神经网络（或共享权重）转化为向量，独立计算。
   * 特点：计算开销低，文档向量可提前离线计算并缓存，适合海量数据召回。
* 交叉编码器（Cross-Encoder / Re-ranker）：
* 逻辑：将查询词和目标文档拼接成一个长文本（如 [CLS] Query [SEP] Document），同时喂入大模型，让注意力机制（Attention）充分捕获每一对字词的深度交互。
   * 特点：准确度极高，但计算极其昂贵。通常不用于第一路召回，而用于最后的精排（Re-ranking）。
* 

## 2. 近似最近邻（ANN）路由算法
向量检索的瓶颈在于海量高维向量的距离比对，因此底层必须依赖高效的索引算法：

* HNSW（分层导航小世界）：通过模仿“跳表”构建多层图拓扑结构，实现 $O(\log N)$ 复杂度的极速路由，是目前主流向量数据库（如 Chroma、Milvus）的首选。
* IVF（倒排文件索引）：利用 K-Means 将空间聚类，检索时只对比最近的几个簇，配合 PQ（乘积量化） 压缩技术，极其适合超大规模（十亿级）的内存节约型场景。
* 

------------------------------
## 三、 当前前沿：混合检索与融合重排（Hybrid Search & Fusion）
在生产级别的 RAG 系统中，任何单一的检索方式都有硬伤。当前的行业最佳实践是混合检索架构。

                    ┌──> BM25 检索 (字面精准) ──> 列表A ──┐
用户 Query ───┤                                     ├──> RRF (倒数秩融合) ──> 统一 Top-N ──> Cross-Encoder 精排 ──> LLM
                    └──> 向量检索 (语义泛化) ──> 列表B ──┘


* 混合召回（Multi-Route Retrieval）：同时运行 BM25 文本检索 和 稠密向量检索。
* RRF（倒数秩融合）算法：一种无视原始分数尺度的重排算法。它只根据文档在各路召回中的排名（Rank）倒数进行累加，完美融合字面与语义的检索结果。
* SPLADE（稀疏向量大模型检索）：一种新兴的折中方案。利用大模型预测词表的权重，输出一个包含语义扩展的稀疏向量，既保留了倒排索引的高效，又具备了语义泛化能力。

------------------------------
## 信息检索算法全景对比

| 算法 / 技术      | 检索类型 | 核心关注点                 | 适用场景                         | 局限性                         |
| ---------------- | -------- | -------------------------- | -------------------------------- | ------------------------------ |
| BM25             | 稀疏检索 | 关键词字面匹配、词频饱和度 | 查特定型号、人名、日志、精确编号 | 无法识别同义词、错别字         |
| HNSW (Embedding) | 稠密检索 | 语义空间距离（余弦、点积） | 问答系统、跨语言检索、意图理解   | 容易产生语义幻觉，内存消耗巨大 |
| Cross-Encoder    | 深度重排 | 词与词之间的全注意力交互   | 检索完成后的 Top-10 精准重排     | 计算极慢，无法用于首轮海量检索 |
| RRF              | 结果融合 | 各路检索结果的相对排名     | 混合检索（Hybrid）的结果合并     | 需要依赖上游多路检索质量       |
```

# 基于字面匹配的稀疏检索（Sparse Retrieval）

## BM25

- BM25 文本检索原理
```markdown
BM25（Best Matching 25） 是一种经典的基于概率检索框架的稀疏向量文本检索算法。它是 TF-IDF 算法的升级版，也是目前 Elasticsearch、Lucene 等主流搜索引擎计算文本相关性的核心算法。 [1, 2, 3] 
其核心任务是：给定一个用户的搜索词（Query），计算它与目标文档（Document）之间的相关性得分，并以此进行排序。 [4] 
------------------------------
## 1. BM25 核心数学公式
假设用户输入的查询词为 $Q$，经过分词后得到一组关键词 $\{q_1, q_2, ..., q_n\}$。文档 $D$ 与查询 $Q$ 的 BM25 相关性得分计算公式如下： [1, 5, 6, 7] 
$$Score(D, Q) = \sum_{i=1}^{n} IDF(q_i) \cdot \frac{f(q_i, D) \cdot (k_1 + 1)}{f(q_i, D) + k_1 \cdot \left(1 - b + b \cdot \frac{\vert{}D\vert{}}{avgdl}\right)}$$ 
公式中各组件含义：

* $q_i$：查询词中的第 $i$ 个关键词。
* $IDF(q_i)$：关键词 $q_i$ 的逆文档频率，用来衡量该词的稀缺程度和核心权重。
* $f(q_i, D)$：关键词 $q_i$ 在当前文档 $D$ 中出现的词频（TF）。
* $\vert{}D\vert{}$：当前文档 $D$ 的字数/词数（文档长度）。
* $avgdl$（Average Document Length）：整个文档库中所有文档的平均长度。
* $k_1$ 与 $b$：调节算法行为的超参数。
* [1, 5, 8, 9] 

------------------------------
## 2. 三大核心原理剖析
BM25 成功的关键在于它通过三个维度对文档进行了精准的权重校准： [10] 
## ① 逆文档频率（IDF）—— 评估词的重要程度

* 原理：如果一个词在全网所有文档中都频繁出现（如“的”、“什么是”、“包含”），那它就无法代表用户的检索意图；相反，如果一个词非常罕见（如“大模型”、“RRF算法”），那它就具备极高的区分度。
* 计算：BM25 通常采用经过平滑处理的经典 IDF 算法：
$$IDF(q_i) = \ln \left( \frac{N - n(q_i) + 0.5}{n(q_i) + 0.5} + 1 \right)$$ 
（其中 $N$ 为文档总数，$n(q_i)$ 为包含关键词 $q_i$ 的文档数。）
* [5, 7, 8, 10] 

## ② 词频饱和度（TF Saturation）—— 防止作弊与过度加权

* TF-IDF 的缺陷：在传统 TF-IDF 中，词频与得分呈线性关系。如果一篇文档故意将“苹果”重复 1000 次（关键词堆砌），它的得分会高得离谱。
* BM25 的改进：引入超参数 $k_1$ 控制词频增长的上限（默认值通常为 $1.2$ 到 $2.0$）。
* 效果：当词频 $f(q_i, D)$ 增加时，分数的增长曲线会逐渐平缓并趋于饱和。一个词在文章中出现 5 次和出现 50 次，带给 BM25 的权重增幅非常微弱，有效抑制了“关键词堆砌”。
* [1, 9, 10, 11, 12, 13] 

## ③ 文档长度归一化（Length Normalization）—— 公平对待长短文本

* 长文档的优势缺陷：长文章（如万字长文）由于字数多，天然更容易包含某个关键词，但这并不意味着它比一篇只有 200 字但字字精辟的短文更相关。
* BM25 的改进：引入超参数 $b$（默认值通常为 $0.75$）和比例 $\frac{\vert{}D\vert{}}{avgdl}$。
* 若 $b=1$，表示完全对文档长度进行惩罚。如果文章长度是平均长度的两倍，其词频贡献就会被严重稀释。
   * 若 $b=0$，表示完全不考虑文档长度的影响。
* 效果：通过测量当前文档与平均文档长度的比例，对长文档的词频进行适度“打折”，确保短文档也能获得公平的竞争机会。 [1, 2, 8, 10, 12] 

------------------------------
## 3. BM25 与 向量检索（Embedding）的对比

| 特性 [1, 3, 7, 14] | BM25 文本检索                              | 向量检索（Dense Embedding）                        |
| ------------------ | ------------------------------------------ | -------------------------------------------------- |
| 检索机制           | 严格的关键词字面匹配（精确匹配）           | 语义空间距离计算（模糊匹配）                       |
| 优点               | 对人名、型号、专有名词、错误拼写极其精准   | 能够理解同义词、上下文和隐含的意图                 |
| 缺陷               | 无法理解同义词（如“西红柿”匹配不到“番茄”） | 面对罕见专有名词、版本号（如“V2.4”）时容易产生幻觉 |
| 计算开销           | 极低（基于倒排索引，速度极快）             | 较高（需要 GPU 转换及向量数据库的 ANN 搜索）       |

工业界最佳实践：在构建高可用的 RAG（检索增强生成）系统时，通常不会单独使用其中一种。而是使用 BM25 负责字面硬匹配 + 向量检索负责语义软匹配，最后通过 RRF（倒数秩融合）算法 将两路召回的结果进行重排融合。
```

# 基于深度学习的稠密检索（Dense Retrieval）

- 基于向量的语义检索原理和算法
```markdown
基于向量的语义检索（Dense Retrieval） 突破了传统 BM25 算法仅依赖“字面匹配”的局限。它通过深度学习模型将文本转化为高维密集向量，在“语义空间”中计算文本之间的距离，从而能够识别同义词、上下文语境和隐含的意图。
------------------------------
## 1. 核心流程与原理
语义检索的核心思想是：语义越相似的文本，在多维数学空间中的距离就越近。其标准工作流包含以下三个步骤：
## ① 向量化（Embedding）
利用预训练的双塔模型（Bi-Encoder，如 embeddinggemma、text-embedding-3-small），将离散的文本符号转化为固定维度的实数向量。

* 文档端（Offline）：将海量文档切块（Chunks），提前通过模型计算出向量，存入向量数据库。
* 查询端（Online）：用户输入查询词（Query），实时通过同一个模型转化为同维度的向量。

## ② 空间度量（Distance Metrics）
在多维空间中，通过数学公式计算两个向量之间的相似度。最常用的度量指标包括：

* 余弦相似度（Cosine Similarity）：关注向量夹角，不关注绝对长度。适合文本长度不一的场景。公式：$\cos(\theta) = \frac{A \cdot B}{\Vert{}A\Vert{} \Vert{}B\Vert{}}$。
* 点积/内积（Dot Product）：如果向量已经过单位化（L2 归一化），点积等价于余弦相似度，且计算速度极快。
* 欧氏距离（L2 Distance）：计算两点间的绝对物理距离。数值越小，相似度越高。

## ③ 近似最近邻搜索（ANN）
如果针对数百万个向量进行逐一对比（暴力搜索 KNN），计算复杂度为 $O(N)$，在生产环境中无法做到毫秒级响应。因此，工业界引入了近似最近邻（Approximate Nearest Neighbor, ANN）算法，在极小的精度损失下，将检索时间缩短至 $O(\log N)$ 或 $O(1)$。
------------------------------
## 2. 核心 ANN 路由算法
向量数据库（如 Chroma、FAISS、Milvus）底层依赖不同的 ANN 算法来实现极速检索，主要分为以下三类：
## ① 基于图（Graph-based）—— 当前的主流标准：HNSW
HNSW（Hierarchical Navigable Small World，分层导航小世界） 是目前工业界效果最好、使用最广的向量索引算法。

* 跳表结构：它借鉴了数据结构中“跳表”的思想，将向量图分为多层。
* 高速路由：
* 在顶层（稀疏图）进行大跨度的粗粒度搜索，快速定位到目标大致区域。
   * 逐层向下钻取，在底层（密集图）进行精细化的局部搜索。
* 优缺点：检索速度极快，召回率极高。缺点是构建索引时非常消耗内存（RAM）。

## ② 基于聚类/倒排（Inverted File）—— 适合超大规模：IVF
IVF（Inverted File Index，倒排文件索引） 试图通过对空间进行切分来减少对比次数。

* K-Means 聚类：在初始化时，使用 K-Means 算法将全量向量聚类成 $K$ 个簇（聚类中心）。
* 两阶段检索：
* 第一阶段：计算查询向量与这 $K$ 个聚类中心的距离，找出最接近的 $n$ 个簇。
   * 第二阶段：仅在这 $n$ 个簇内部进行向量的逐一对比。
* 配合 PQ 压缩：通常与乘积量化（Product Quantization, PQ）结合。PQ 将 1024 维的高精浮点数压缩为低字节的编码，大幅削减显存/内存占用，适合海量数据。

## ③ 基于树（Tree-based）—— 经典但逐渐边缘化：KD-Tree / Annoy

* 原理：通过超平面不断将高维空间一分为二，构建二叉树。检索时顺着树枝向下查找。
* 局限：在维度较低（如几维到几十维）时效率极高，但面对现代 Embedding 模型（通常为 768 或 1536 维）时，会陷入“维度灾难”，性能退化至与暴力搜索无异。

------------------------------
## 3. 语义检索的硬伤与优化
虽然语义检索能完美解决“西红柿 -> 番茄”的同义词问题，但在实际企业级 RAG 应用中存在以下天生缺陷：

   1. 对专有名词不敏感：例如输入产品型号“MacBook Pro M4 2026”，由于“M4”和“2025”在语义空间中离“M3”或“2026”极近，模型极易召回错误版本的文档（即发生语义幻觉）。
   2. 长文本语义稀释：若 Chunk 切得过大，一句话的独特性会被长篇大论稀释，导致精准度下降。

工业界落地解法：
正如前文所述，为了结合 BM25（专有名词精准） 与 向量检索（语义模糊泛化） 的双重优势，标准架构是运行双路召回，并使用 RRF 算法 将结果合并，最后交给大模型。
```

- HNSW（Hierarchical Navigable Small World，分层导航小世界）原理
```markdown
HNSW（Hierarchical Navigable Small World，分层导航小世界） 是目前工业界公认性能最顶尖的近似最近邻（ANN）向量检索算法。它巧妙地结合了数据结构中的跳表（Skip List）思想与图论中的小世界网络（Small World Network）拓扑结构，成功将高维向量的搜索复杂度从暴力搜索的 $O(N)$ 降低到了惊人的 $O(\log N)$。
要彻底理解 HNSW，我们需要将其拆解为三个核心演进阶段：
------------------------------
## 1. 基石一：NSW（导航小世界图）
在 HNSW 出现之前，先有了 NSW（Navigable Small World）。

* 构图机制：当新向量节点插入时，通过邻近搜索，将其与空间中最近的 $M$ 个既有节点相连。
* 网络特性：这样构建出来的图具有“小世界”特性——高聚集度（局部有很多紧密相连的小圈子）和短平均路径长度（虽然绝大多数节点不直接相连，但通过少数“远距离长链桥梁”，任意两点间只需很少的跳数即可到达）。
* 检索方式（贪心搜索）：从图中的某一个入口节点（Entry Point）开始，计算当前节点的所有邻居与查询向量（Query）的距离。如果发现某个邻居比当前节点更接近 Query，就跳过去。重复此过程，直到所有邻居都回传更远的距离，则当前节点即为局部最优解。
* NSW 的致命缺陷：
* 容易陷入局部最优：由于缺乏全局视角的引导，搜索很容易卡在某个局部死胡同。
   * 大体量下效率降低：当图中的节点达到千万级时，贪心搜索在密集的局部网络中会产生大量的“冗余计算”，导致检索速度显著下滑。

------------------------------
## 2. 基石二：跳表（Skip List）思想的引入
为了解决 NSW 在大数据量下的路由效率问题，HNSW 引入了多层结构，其灵感完全来源于一维数据结构中的跳表：

第 2 层 (最稀疏) ->  [1] -------------------------> [9] -------------------------> [17]
第 1 层 (较稀疏) ->  [1] ---------> [5] ---------> [9] ---------> [13] ---------> [17]
第 0 层 (最密集) ->  [1]->[3]->[5]->[7]->[9]->[11]->[13]->[15]->[17]


* 在普通链表中查询一个数需要 $O(N)$ 时间。
* 跳表通过在上方构建多层“稀疏索引”，让指针可以实现大跨度的“飞跃”。检索时从顶层开始粗定位，向下钻取，最后在底层完成精准定位，时间复杂度降为 $O(\log N)$。

------------------------------
## 3. HNSW 的完整工作原理
HNSW 将“跳表的多层架构”完美嫁接到了“NSW 的图结构”上。它把向量空间切分为由上到下的多个图层（Layers）：

第 2 层 (高速公路)   [节点A] ---------------------------- [节点B]  (大跨度粗定位)

                       |                                    |
第 1 层 (省道网络)   [节点A] ------- [节点C] ------------- [节点B]  (中等跨度导航)

                       |               |                    |
第 0 层 (城市街道)   [节点A]-[节点D]-[节点C]-[节点E]-[节点F]-[节点B]  (底层全量密集搜索)

## ① 数据结构设计

* 第 0 层（Bottom Layer）：包含向量数据库中的所有向量节点，它们之间构建成一个极其密集的 NSW 图。
* 上层（Upper Layers）：层数越高，节点越稀疏。每个节点被分配到更高层的概率遵循指数衰减概率分布。
* 纵向连接：同一个向量如果在多层中存在，它们在垂直方向上是完全对齐且相互锚定的。

## ② 检索（Search）流程
当用户输入一个查询向量 $Query$，要求寻找最相似的 $K$ 个结果时：

   1. 顶层切入：从系统预设的最高层（最稀疏层）的唯一入口点（Enter Point）开始。
   2. 高层高速冲刺：在当前层执行贪心搜索，利用稀疏图的大跨度长链快速向 $Query$ 逼近。当在当前层找不到更近的邻居时，停止搜索。
   3. 垂直下钻：将当前层找到的那个“局部最近点”作为下一层的入口点，降到下一层。
   4. 逐层逼近：在较密集的下一层继续重复贪心搜索，定位到更新的最近点，再下钻。
   5. 底层精准收网：降落到第 0 层时，检索算法已经距离 $Query$ 非常近了。此时，算法会在第 0 层高密度网络中维护一个动态队列（由参数 efSearch 控制大小），进行精细化的邻居扩展，最终输出距离最近的 $K$ 个向量。

------------------------------
## 4. HNSW 的关键控制参数
在实际使用（如初始化 FAISS 或 Milvus 索引）时，你会遇到以下三个最核心的调优参数：

* M（每个节点的最大邻居数）：控制图中每个节点可以连接多少条边。
* $M$ 越小，内存占用越少，构图越快，但高维空间的连接性变差，召回率降低。
   * 通常取值范围：8 到 64。
* efConstruction（构建时探索深度）：控制建图（Index Building）时，为了给新节点找邻居而遍历的节点数量。
* 该值越大，建图时间成倍增加，但图的拓扑结构越完美，检索召回率越高。
   * 通常取值范围：100 到 500。
* efSearch（检索时探索深度）：在第 0 层检索时，动态候选队列的大小。
* 这是一个可以在运行时调节的参数。$efSearch$ 越大，检索越慢，但召回率（准确率）越高。
   * 必须满足 $efSearch \ge K$（$K$ 为要求的返回结果数）。

## 5. 优缺点总结

* 绝对优势：检索速度（QPS）和高召回率（Recall）在目前所有 ANN 算法中几乎无出其右，对动态增删节点的支持也比较友好。
* 核心劣势：内存吞吐巨兽。由于需要将所有多层图结构以及向量数据全部加载进内存（RAM/显存），并且指针边关系非常消耗空间，它的内存开销通常是原始向量文件大小的数倍。
```

# 混合检索与融合重排（Hybrid Search & Fusion）

- 倒数秩融合（RRF）算法
```markdown
倒数秩融合（Reciprocal Rank Fusion, RRF） 是一种在信息检索和 RAG（检索增强生成）系统中常用的多路召回重排算法。
它用于将来自不同检索渠道（如：BM25 文本检索和基于向量的语义检索）的结果列表，合并为一个统一的、按相关性排序的最终列表。
## 1. 核心数学公式
RRF 的核心思想是：一个文档在各个检索结果列表中的排名（Rank）越靠前，它的最终得分就越高。
对于文档集合中的某一个文档 $d$，其 RRF 得分计算公式为：
$$RRF\_Score(d) = \sum_{m \in M} \frac{1}{k + r_m(d)}$$ 

* $M$: 参与融合的检索渠道集合（例如：$M = \{\text{向量检索}, \text{关键字检索}\}$）。
* $r_m(d)$: 文档 $d$ 在渠道 $m$ 中的排名（从 1 开始计数）。如果文档未出现在该渠道的 Top-N 结果中，则该项得分为 0。
* $k$: 一个常量（常数惩罚项），主要用于平滑分数，防止排名靠前的文档权重过高。默认经验值通常设为 60。

## 2. 核心优势

* 无需归一化：不同检索算法的分数区间（如 BM25 可能是 0~30，向量相似度是 0~1）无法直接相加。RRF 只看排名，不看原始分数，完美解决了尺度不一的问题。
* 实现极其简单：不需要训练任何机器学习模型，纯数学计算，运行速度极快。
* 效果稳健：即使某个检索渠道的效果很差，RRF 也不会轻易被噪声污染，具有很强的鲁棒性。

## 3. 具体计算示例
假设我们要融合两个渠道（向量检索、BM25 检索）的结果，设置 $k = 60$：

* 文档 A：向量检索排第 1 名，BM25 排第 4 名。
$$RRF\_Score(A) = \frac{1}{60 + 1} + \frac{1}{60 + 4} = \frac{1}{61} + \frac{1}{64} \approx 0.01639 + 0.01562 = 0.03201$$ 
* 文档 B：向量检索排第 2 名，BM25 未上榜。
$$RRF\_Score(B) = \frac{1}{60 + 2} + 0 = \frac{1}{62} \approx 0.01613$$ 

结果：文档 A 的最终得分高于文档 B，因此在融合后的总列表中排在最前面。
## 4. Python 代码实现
你可以通过以下简单的 Python 代码手动实现 RRF 算法：

from collections import defaultdict
def reciprocal_rank_fusion(results_lists, k=60):
    """
    results_lists: 包含多个渠道检索结果的列表，例如 [['doc_a', 'doc_b'], ['doc_c', 'doc_a']]
    k: 平滑常数，默认 60
    """
    rrf_scores = defaultdict(float)
    
    # 遍历每一个检索渠道的结果列表
    for results in results_lists:
        for rank, doc_id in enumerate(results, start=1):
            # 累加倒数秩得分
            rrf_scores[doc_id] += 1.0 / (k + rank)
            
    # 按得分从高到低排序
    sorted_docs = sorted(rrf_scores.items(), key=lambda x: x[1], reverse=True)
    return sorted_docs
# 测试数据vector_results = ["doc_1", "doc_2", "doc_3"]bm25_results = ["doc_3", "doc_1", "doc_5"]
final_rank = reciprocal_rank_fusion([vector_results, bm25_results], k=60)
print(final_rank)
```
